AC21.01

Concevoir un réseau informatique adapté à une petite structure

Ce que j'ai fait

Conception et déploiement d'une infrastructure réseau complète (adressage IP, VLANs, routage statique) pour simuler une petite entreprise.

Pourquoi

Répondre aux besoins de cloisonnement et de sécurité d'une structure multiservice, en appliquant les bonnes pratiques d'architecture.

Comment

Utilisation de Cisco Packet Tracer pour la maquette, configuration des VLANs sur switchs managés, rédaction d'un plan d'adressage documenté.

Difficultés

La gestion des VLANs entre plusieurs switchs (trunk vs access) a nécessité plusieurs reconfiguration avant d'obtenir une communication inter-VLAN fonctionnelle.

Ce que j'en ai appris

L'importance de tester chaque VLAN indépendamment avant d'interconnecter les équipements, et de toujours documenter le plan d'adressage avant de commencer.

Ce que je ferais autrement

J'aurais utilisé un template de documentation dès le début plutôt que de rédiger le schéma a posteriori, pour gagner du temps sur les phases de débogage.

AC21.02

Déployer des services réseau (DNS, DHCP, services avancés)

Ce que j'ai fait

Installation et configuration d'un serveur DHCP, DNS et d'un service de supervision sur une VM Linux (Debian).

Pourquoi

Ces services sont fondamentaux pour l'administration de tout réseau d'entreprise. Les maîtriser en installation manuelle permet d'en comprendre le fonctionnement interne.

Comment

Utilisation de isc-dhcp-server et bind9 sous Debian, configuration des fichiers /etc/dhcp/dhcpd.conf et zones DNS, tests via dig et nslookup.

Difficultés

Les erreurs de syntaxe dans le fichier de zones DNS étaient silencieuses — bind9 refusait de démarrer sans message d'erreur explicite, ce qui a rendu le débogage difficile.

Ce que j'en ai appris

L'outil named-checkconf et named-checkzone permettent de valider la configuration avant redémarrage. Cette pratique de "lint avant déploiement" est désormais systématique dans ma méthode.

Ce que je ferais autrement

Je mettrais en place un environnement de test distinct du réseau de production pour tester les modifications de configuration sans risque d'interruption.

AC22.01

Mesurer et analyser des signaux dans un système de transmission

Ce que j'ai fait

Analyse de signaux avec un analyseur de spectre et des outils logiciels (Matlab Simulink) pour caractériser des transmissions.

Pourquoi

Comprendre le comportement fréquentiel des signaux est indispensable pour diagnostiquer des problèmes de transmission et choisir les bons filtres.

Comment

Mesures en laboratoire, export des données vers Matlab pour post-traitement, comparaison des spectres théoriques et mesurés.

Difficultés

La conception de filtres numériques sous Simulink a posé problème : les paramètres de coupure ne correspondaient pas aux résultats attendus, révélant une méconnaissance des fenêtres d'analyse.

Ce que j'en ai appris

La différence entre les fenêtres de Hamming, Hanning et rectangulaire et leurs effets sur la précision spectrale. Je comprends maintenant pourquoi le choix de la fenêtre n'est pas anodin.

Ce que je ferais autrement

Je validerais d'abord le modèle avec un signal test simple et connu avant de passer aux signaux réels, pour isoler les erreurs de modélisation des erreurs de mesure.

AC22.02

Déployer et configurer un accès sans-fil sécurisé

Ce que j'ai fait

Mise en place d'un réseau Wi-Fi d'entreprise avec authentification WPA2-Enterprise (RADIUS), séparation des SSIDs par profil utilisateur.

Pourquoi

Un réseau Wi-Fi d'entreprise ne peut pas fonctionner avec une simple clé PSK partagée : chaque utilisateur doit avoir des identifiants propres, traçables et révocables.

Comment

Configuration d'un serveur FreeRADIUS, intégration avec Active Directory via LDAP, paramétrage des points d'accès Cisco en mode controller.

Difficultés

La chaîne RADIUS → LDAP → AD présentait des problèmes d'authentification intermittents liés aux timeouts. Le débogage nécessitait de lire les logs à trois niveaux simultanément.

Ce que j'en ai appris

L'importance d'une approche systématique dans le débogage des systèmes distribués : tester chaque maillon indépendamment (RADIUS seul, LDAP seul) avant de tester la chaîne complète.

Ce que je ferais autrement

J'aurais mis en place une supervision des logs en temps réel dès le début plutôt que de consulter les fichiers manuellement lors de chaque test.

AC23.01

Développer des scripts d'automatisation système

Ce que j'ai fait

Développement de scripts Bash et Python pour automatiser des tâches d'administration : inventaire réseau, sauvegarde de configurations, génération de rapports.

Pourquoi

Les tâches répétitives d'administration sont sources d'erreurs humaines et chronophages. L'automatisation garantit une exécution fiable et reproductible.

Comment

Python avec paramiko pour les connexions SSH, netmiko pour l'automatisation des équipements réseau Cisco, structuration en fonctions réutilisables.

Difficultés

La gestion des erreurs réseau (timeout, équipements non joignables) n'était pas prévue initialement, ce qui rendait les scripts fragiles en environnement réel.

Ce que j'en ai appris

Un script "qui marche en lab" n'est pas prêt pour la production. La gestion des exceptions et les mécanismes de retry sont aussi importants que la fonctionnalité principale.

Ce que je ferais autrement

Je commencerais par les cas d'échec avant de coder le chemin nominal — une approche défensive qui m'aurait économisé plusieurs heures de débogage.

📎 Trace : extrait de script Python
import netmiko, json
from netmiko import ConnectHandler

def get_running_config(host, user, pwd):
    try:
        conn = ConnectHandler(
            device_type='cisco_ios',
            host=host, username=user, password=pwd
        )
        config = conn.send_command('show running-config')
        conn.disconnect()
        return config
    except Exception as e:
        print(f"[ERREUR] {host}: {e}")
        return None
AC23.02

Développer et déployer une application web pour un service réseau

Ce que j'ai fait

Développement d'un site portfolio avec backend Flask, base de données SQL et déploiement via Docker sur un VPS.

Pourquoi

Comprendre la chaîne complète de déploiement d'une application web — du développement à la mise en production — est une compétence essentielle pour administrer des services d'entreprise.

Comment

Stack Flask + SQLite, conteneurisation Docker avec docker-compose, reverse proxy Nginx pour l'exposition HTTPS, scripts de déploiement automatisé.

Difficultés

La configuration de la communication inter-conteneurs Docker (réseau bridge, DNS interne) n'était pas intuitive. Le conteneur Flask ne trouvait pas le conteneur de base de données par son nom.

Ce que j'en ai appris

Avec docker-compose, les services sont accessibles par leur nom de service défini dans le fichier YAML, pas par l'IP. La documentation officielle Docker Compose est un préalable indispensable.

Ce que je ferais autrement

Je passerais directement à Docker Compose dès le début plutôt que de gérer des conteneurs individuels avec docker run, l'orchestration étant nécessaire dès qu'il y a plusieurs services.

AC24.01

Identifier les vulnérabilités d'un SI et proposer des remédiations

Ce que j'ai fait

Audit de sécurité d'un réseau de test : scan de ports avec Nmap, analyse des services exposés, identification des CVE associées et proposition d'un plan de remédiation priorisé.

Pourquoi

Avant de sécuriser un système, il faut d'abord en connaître la surface d'attaque réelle. Une sécurité perçue sans audit régulier est une fausse sécurité.

Comment

Nmap pour la reconnaissance, Nessus (essai) pour l'analyse des vulnérabilités, croisement avec la base CVE du NIST, rédaction d'un rapport structuré avec scores CVSS.

Difficultés

La priorisation des vulnérabilités est complexe : un score CVSS élevé ne signifie pas nécessairement un risque élevé dans un contexte donné. Il faut croiser avec l'exposition réelle du service.

Ce que j'en ai appris

La différence entre vulnérabilité et risque : une faille critique sur un service non exposé à Internet est moins prioritaire qu'une faille modérée sur un service public. Le contexte prime sur le score brut.

Ce que je ferais autrement

Je documenterais mieux le périmètre d'audit avant de commencer pour éviter de scanner des systèmes hors-scope — une erreur qui peut avoir des conséquences légales.

AC24.02

Sensibiliser des utilisateurs aux risques de cybersécurité

Ce que j'ai fait

Conception et animation d'une présentation de sensibilisation aux risques numériques pour simuler le rôle de RSSI face à des personnels non-techniques.

Pourquoi

La technique seule ne suffit pas : 90% des incidents de sécurité commencent par une erreur humaine. Former les utilisateurs est souvent plus efficace que d'ajouter des pare-feux.

Comment

Modules MOOC ANSSI comme base documentaire, présentation sur les 4 bonnes pratiques clés, scénarios concrets d'attaques (phishing, ingénierie sociale), travail en binôme sur un mois.

Difficultés

Adapter le niveau technique au public : trop technique, on perd l'audience ; trop simplifié, les messages ne sont pas retenus. Trouver le bon équilibre a demandé plusieurs itérations.

Ce que j'en ai appris

La pédagogie est une compétence à part entière. Utiliser des exemples concrets et des anecdotes réelles rend les messages de sécurité beaucoup plus mémorables que des listes de règles.

Ce que je ferais autrement

J'inclurais un quiz ou un exercice de simulation d'hameçonnage pour mesurer l'impact réel de la sensibilisation plutôt que de me fier aux retours subjectifs des participants.

AC25.01

Mettre en place des outils de supervision réseau

Ce que j'ai fait

Déploiement d'une solution de monitoring réseau (Zabbix / Grafana) pour surveiller la disponibilité et les performances d'une infrastructure.

Pourquoi

La supervision proactive permet de détecter les problèmes avant qu'ils deviennent des incidents et de disposer de données historiques pour analyser les tendances.

Comment

Installation de Zabbix server sur VM, agents SNMP sur les équipements réseau, création de dashboards Grafana via l'API Zabbix, configuration d'alertes par seuil.

Difficultés

La configuration SNMP sur les switchs Cisco nécessite une correspondance exacte entre la communauté déclarée sur l'équipement et celle configurée dans Zabbix — une erreur subtile qui bloque toute collecte.

Ce que j'en ai appris

La valeur des données de supervision ne réside pas dans leur collecte mais dans leur interprétation. Définir des seuils pertinents demande de connaître le comportement normal du réseau.

Ce que je ferais autrement

Je commencerais par établir une baseline de métriques normales sur 2 semaines avant de configurer les alertes, pour éviter les faux positifs qui conduisent à ignorer les vraies alertes.
